# Fixed & Resolved Issues (Chapters 1 to 11)

This document tracks all the architecture, storage, performance, and security issues resolved across Chapters 1 through 11 of the Osdag-Web codebase.

| # | Chapter / Issue Title | Target Component | Description (1-2 lines) | Resolution Summary |
|---|------------------------|------------------|--------------------------|--------------------|
| **1** | **Chapter 1: Resource Waste from HTTP Polling** | WebSocket / Channels | Polling calculation status at 1s intervals generated significant HTTP overhead. | Transitioned to a real-time WebSocket push model using Django Channels and Redis broadcasts. |
| **2** | **Chapter 1: Celery Result Bloat in Redis** | Task Queue / Redis | Finished task payloads remained indefinitely in Redis memory. | Set `CELERY_RESULT_EXPIRES = 1800` to automatically purge finished task payloads after 30 minutes. |
| **3** | **Chapter 2: Storage Accumulation & Memory Leaks** | File Storage / Cleanup | CAD and PDF generation saved physical files permanently, causing disk space exhaustion. | Implemented in-memory streaming for downloads and a daily Celery Beat task to delete files older than 24 hours. |
| **4** | **Chapter 2: Redundant Debug and Dead Code** | Backend Adapters | Backend module adapters contained dead stdout redirection blocks and duplicate `validate_input_new` validators. | Cleaned up all redundant stdout redirection lines and duplicate validation definitions across all module adapters. |
| **5** | **Chapter 2: Multi-Node Distributed Worker Challenges** | Docker Compose / volumes | Isolated container filesystems prevented Django from serving CAD files written locally by Celery workers. | Mounted a shared named Docker volume `file_storage_volume` across all containers on the baremetal host. |
| **6** | **Chapter 3: Performance Overhead of JWT Verification** | Auth Middleware | RSA verification of Firebase ID tokens on every request added ~15-50ms latency. | Cached verified token metadata in Redis with dynamic TTL matching the token's lifetime, reducing lookup to <1ms. |
| **7** | **Chapter 3: Logical Email Verification Bypasses** | API Permissions / Auth | Endpoints bypassed email verification for `inline` exports or lacked verified check constraints. | Enforced DRF custom permissions globally (`IsEmailVerifiedIfAuthenticated`) and on all project views (`IsEmailVerified`). |
| **8** | **Chapter 4: Legacy module_id Compatibility & Mapping Redundancy** | Project API & Frontend | Redundant `module_id` response attributes and multiple fragmented mappings for naming variations existed in the frontend. | Removed `module_id` from project views, replaced it with standard `submodule`, and unified name mapping via a central helper. |
| **9** | **Chapter 4: High DB Write Volume on Auto-save** | Design Submission / DB | Saving the full project payload on every successful calculation created heavy PostgreSQL write traffic. | Implemented a 5-second debounced save on calculation success, with immediate flush on unmount to prevent data loss. |
| **10** | **Chapter 4: Project Creation Before Calculation Run** | Project Creation Flow | Projects could be registered with inputs configured but outputs left as `null`, bypasses design verify logic. | Disabled the "Create Project" button and blocked keyboard shortcut actions if calculations have not run (`hasOutput` is false). |
| **11** | **Chapter 4: Legacy & Unimplemented Database Schema Fields** | Backend Models / DB Schema | Legacy `Design` model, `Project.osi_file_path`, and `UserAccount.allInputValueFiles` fields/views were dead, relational schema leftovers. | Removed unused database columns, cleaned views/middleware referencing them, deleted `ObtainInputFileView`, and updated database schema. |
| **12** | **Chapter 5: Download OSI in Module Page (DB & Storage Accumulation)** | OSI Export API | Authenticated exports generated physical files on disk and wrote a database entry in `OsiFile`, causing disk bloat. | Streamlined `SaveOsiFromInputs` to always return base64 inline content, avoiding physical file storage and DB write overhead entirely. |
| **13** | **Chapter 5: Download OSI in Project / Dashboard List Page (Key Flattening Drift)** | OSI Export / Flattening | Flattener logic was duplicated and handled manually, leading to key normalization/translation drift. | Integrated dynamic key translation and flattening in `build_osi_payload` on the backend via a unified mappings dict. |
| **14** | **Chapter 5: Import OSI from Homepage / Dashboard (Inconsistent Parsing Engine)** | OSI Homepage Import | Homepage OSI imports used client-side JavaScript (`yaml-js`) which was error-prone and inconsistent with the backend Python parser. | Refactored homepage file loader to route uploaded OSI files through the stateless backend `/api/open-osi/` parser endpoint. |
| **15** | **Chapter 5: Import OSI in Module Page (Key Mapping & Translation Drift)** | OSI Parse / Normalization | Dotted PascalCase keys from flat files had to be mapped and translated entirely on the client-side. | Integrated the translation mappings directly into `parse_osi` on the backend, returning web-ready `{ dock, pref }` inputs. |
| **16** | **Chapter 6: Keyboard Navigation Lock Bypass** | Form Engine / Input Controls | Keyboard-navigating users could press the `Tab` key to focus and edit locked inputs, bypassing the lock interceptor. | Bound text `<input>` and React `<Select>` elements' `disabled`/`isDisabled` props to the `isInputLocked` state. |
| **17** | **Chapter 6: Redundant Network Re-fetches on Custom Section Events** | `useModuleData.js` / Options Fetch | Adding a custom section dispatched `osdag:custom-section-added`, which changed `localCustomSections` state. Because `localCustomSections` was included in the options-fetching `useEffect` dependency array, every custom section registration triggered a full API re-fetch to `/api/options/<module_id>/` even though the server data had not changed. This caused unnecessary HTTP load, re-render cycles, and a complete overwrite of dropdown options on every custom section addition. | Separated the API fetch lifecycle from the client-side merge logic. Removed `localCustomSections` from the `useEffect` dependency array so that the network call fires only on `designType`, `getModuleData`, or `optionsRefetchKey` changes. Introduced a `useMemo` block that re-computes the merged options object on the client whenever either `baseModuleData` (raw API response) or `localCustomSections` changes — without making any network request. |
| **18** | **Chapter 7: Inconsistent Driver Key Preservation (Clobbering `connector_material` on Reset)** | `mergeDesignPrefSyncIntoInputs.js` / Design Pref Merge | `mergeParityIntoInputsPreservingDockDrivers` only protected `["material", "member_material"]` from being overwritten on Defaults/Save. The `connector_material` field, which drives the connector steel grade dropdown across all connection submodules, was missing from the protection list. Clicking Defaults would silently snap the connector material dropdown back to the server default, overriding the user's choice. | Added `"connector_material"` to the `DOCK_DRIVER_KEYS` constant. The preservation loop now restores all three keys from the previous input state after the server merge, preventing any material driver field from jumping on reset. |
| **19** | **Chapter 7: Material Refresh State Races inside the Modal** | `DesignPrefSections.jsx` / Sync Effect | When Save or Defaults completed, `setDesignPrefOverrides` was called after the `await` resolved. Because `syncLoading` and `syncReady` were already committed to their post-operation values (`false`/`true`), the material `useEffect` fired a redundant `runSync("refresh")` in response to the resulting `inputs.connector_material` change, sending an unnecessary network request. | Extended the existing `skipMaterialRefresh` ref (already used to suppress the mount-triggered refresh) to guard managed-operation cascades. `skipMaterialRefresh.current = true` is set immediately before each `setDesignPrefOverrides` call in `saveCoreInputs` and `resetInputs`. The ref is checked synchronously on the triggered re-render, so the effect skips the redundant refresh and resets the flag for genuine future material changes. |
| **20** | **Chapter 8: Global State Cache Mutation** | `GlobalState.jsx` / Catalog Selection | Directly mutating `initialValue.fetch_cache` bypassed React's render cycles, causing cache-clobbering and state race conditions when multiple catalog selectors were mounted. | Transitioned to a React `useRef` (`fetchCacheRef`) to manage the catalog request cache tracking synchronously and safely. |
| **21** | **Chapter 8: Typing-Induced Network Spams in Dependent Data** | `useDependentData.js` / API Fetches | Immediate API request triggers on input parameters created database connection bottlenecks when text was entered rapidly. | Introduced a 250ms debounce delay using `setTimeout` and `clearTimeout` before triggering dependent data network fetches. |
| **22** | **Chapter 8: OSI Prefill Timing Race Hazard** | `useModuleForm.js` / OSI Loader | A fixed 1000ms timeout cleared the prefill session storage before option API calls finished, causing inputs to revert to empty. | Guarded the prefill loading sequence using a options readiness check (`hasLoadedLists`), clearing the cache only after options are successfully populated. |
| **23** | **Chapter 8: Pervasive Debug console.log Statements in Production Code** | Frontend Hooks & State Providers | High-frequency logs (e.g. at page load, CAD render, and design calculation cycles) added rendering overhead and leaked environment parameters. | Cleared legacy debug `console.log` statements from production paths and wrapped development-only diagnostics inside `import.meta.env.DEV` checks. |
| **24** | **Chapter 8: Stale output State Reference in submitDesign Error Handler** | `useDesignSubmission.js` / Error Handler | The error handler checked the React state `output` snapshot, which was stale within the closure, resulting in inaccurate error prompts. | Introduced a local `outputWasSet` tracker to check if calculation outputs were set before any CAD generation exceptions occurred. |
| **25** | **Chapter 8: Unreliable Navigation Release Window in useNavigationGuard** | `useNavigationGuard.js` / Router Guard | An arbitrary 100ms timeout in `performNavigation` caused state update cleanups to run on unmounted components and led to race conditions. | Refactored `allowNavigation` into a React `useRef` to track and release navigation locks synchronously without depending on timeouts. |
| **26** | **Chapter 8: Redundant Legacy Reducer Actions** | `ModuleState.jsx` / `ModuleReducer.jsx` | Redundant deprecated actions (`SAVE_CM_DETAILS`, `SAVE_SDM_DETAILS`, `SAVE_STM_DETAILS`, `UPDATE_SUPPORTING_ST_DATA`, `UPDATE_SUPPORTED_ST_DATA`, and `UPDATE_MATERIAL_FROM_CACHES`) were still dispatched and handled alongside consolidated actions, inflating the bundle. | Refactored `ModuleState.jsx` to dispatch the unified `SAVE_MATERIAL_DETAILS` and `UPDATE_SECTION_DATA` actions, and completely purged all deprecated legacy actions from `ModuleReducer.jsx`. |
| **27** | **Chapter 8: Dead Code: `normalizedFiles` Computation in CAD Submission Path** | `useDesignSubmission.js` / CAD Submission | The `normalizedFiles` mapping logic was computed to normalize lowercase keys to PascalCase but was ignored, as `setCadModelPaths` still read from the raw `cadResult.files`. | Applied `normalizedFiles` to `setCadModelPaths(normalizedFiles)` to ensure normalized keys are stored and used. |
| **28** | **Chapter 8: `resetFormState` Initialization Logic Duplication** | `useModuleForm.js` / Form State Hook | The `resetFormState` function duplicated the identical `.reduce()` initialization blocks from state definitions to reset individual state properties. | Extracted initial state factory helpers (`buildInitialSelectionStates`, `buildInitialAllSelected`, etc.) to be shared by both `useState` initializers and `resetFormState`. |
| **29** | **Chapter 9: WebGL Context Loss Recovery** | `CadViewer.jsx` / R3F Canvas | The R3F Canvas context did not listen to `webglcontextlost` events, causing the canvas to crash on sleep or memory exhaustion. | Added a native listener inside `onCreated` callback that triggers `event.preventDefault()` and increments a state-driven key to force a clean re-mount of the Canvas. |
| **30** | **Chapter 9: Pointer Event Race Hazards** | `SceneManager.jsx` / Hover Handlers | Stale state closures inside R3F pointer-out callbacks caused the hovered state to become locked on components during rapid mouse movements. | Refactored hover callbacks to use the latest ref pattern (`hoveredMeshIdRef`) to read and write current hover state synchronously. |
| **31** | **Chapter 10: Hardcoded UI Status Flags** | `ModulesCardLayout.jsx` / catalog configuration | Submodule availability was hardcoded via string checks (`key === "Truss"` and `label === "PEB"`). | Added `status: "development"` directly to catalog configurations and dynamically checked this attribute in `ModulesCardLayout.jsx` and `SectionCards.jsx`. |
| **32** | **Chapter 10: Incomplete Route Cleanups** | `ModulesCardLayout.jsx` / `SectionCards.jsx` | Unrouted submodules (like `PlateGirder`) caused silent click failures on the homepage. | Configured `PlateGirder` with `status: "development"` to disable/grey out the card, and integrated `react-toastify` in `handleModuleClick` to show toast alerts for any unmapped routes. |
| **33** | **Chapter 11: Hardcoded User ID Permissions** | `docker-compose.yml` / User mapping | Hardcoded user setting `1000:1000` caused file permission issues on developer environments with different UIDs. | Replaced the static string with variable interpolation: `user: "${HOST_UID:-1000}:${HOST_GID:-1000}"`. |
| **34** | **Chapter 11: Static Database Port Mappings** | `docker-compose.yml` / db ports | Static mapping of PostgreSQL to host port 5433 created conflicts with local DB instances. | Updated host port configuration to support environment variable overrides: `ports: - "${DB_HOST_PORT:-5433}:5432"`. |
